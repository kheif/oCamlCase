Bindings

A binding associates a name with a value. In OCaml, let is the only way to introduce a name, and the resulting binding is immutable by default.

The let keyword

The form let <name> = <expr> evaluates the expression on the right and gives the result a name. The name does not refer to a memory cell: it refers directly to the value. There is no assignment, no reassignment, and no notion of a variable in the C sense.

At the top level of a file or a REPL session, a let introduces a name into the surrounding environment. Subsequent code can use that name. The compiler also infers and prints the type, so you immediately see what kind of value you bound.

Key point

A binding is not a variable. It is a fixed link between a name and a value. To get a different value associated with the same name, you write a new let, which produces a new binding.

basic.ml 
(* Bind a name to a value *)
let greeting = "hello"
(* greeting : string *)

let answer = 42
(* answer : int *)

(* Function bindings work the same way *)
let square x = x * x
(* square : int -> int *)

let ... in: local bindings

Inside an expression, let <name> = <expr1> in <expr2> introduces a name that is in scope only within expr2. The whole form is itself an expression and evaluates to whatever expr2 evaluates to.

This is how you give names to intermediate results inside a function body. Each let ... in nests inside the previous one, building up a small local environment.

Local bindings are not visible outside their in clause. After the function returns, the names are gone. This is normal lexical scoping, but it is worth saying out loud because OCaml has no separate var or const form.

local.ml 
let hypotenuse a b =
  let a2 = a *. a in
  let b2 = b *. b in
  Float.sqrt (a2 +. b2)

(* a2 and b2 are scoped to the body.
   They do not exist outside hypotenuse. *)

Shadowing

Writing a second let with the same name does not mutate the original binding. It introduces a new binding that shadows the old one. Code written before the shadow still sees the original value; code after sees the new one.

This is how you express "update" in a language without assignment. The old value is unchanged; you just stop referring to it.

shadow.ml 
let x = 1
let show_first () = x  (* captures the original 1 *)

let x = 2           (* shadows: a new binding for x *)

show_first ()        (* still 1, not 2 *)
x                    (* 2 *)

Mutability is opt-in

If you genuinely need a value that changes over time, OCaml gives you ref. A ref is a small mutable cell. The binding to the cell is still immutable, but the cell's contents can be updated with := and read with !.

In practice, idiomatic OCaml leans on immutable bindings and recursion or higher-order functions for what other languages do with loops and reassignment. Reach for ref only when you truly need state.

refs.ml 
let counter = ref 0
(* counter : int ref *)

counter := !counter + 1
counter := !counter + 1

!counter             (* 2 *)

(* The binding `counter` still points to the same cell.
   What changed is the cell's contents, not the binding. *)
Good practice

Default to plain let bindings. They are easier to reason about, friendlier to the type checker, and avoid the class of bugs that come with shared mutable state.
