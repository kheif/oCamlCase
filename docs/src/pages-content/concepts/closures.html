Concept 5 of 14

Closures

A function value is more than its code. To run later, it also needs to remember the meaning of every name it uses freely. The pair of code and remembered environment is called a closure, and OCaml builds one every time you write fun or let f x = ....

Environments

An environment is a collection of identifier bindings. The notation [x := 5, y := 7] means the name x stands for the value 5 and y stands for 7. Given an environment, every closed expression has a value. x + y in this environment evaluates to 12.

An identifier that has no binding in the surrounding environment is called free. A declaration with free identifiers cannot be executed on its own, because the runtime would not know what those names stand for. let q x = 3 + p x only makes sense if p is already bound somewhere outside.

Key point

The same expression can have different values in different environments. The environment is part of the meaning of a phrase, not just a runtime detail.

env.ml 
(* Environment [x := 5, y := 7] *)
let x = 5
let y = 7

(* In this environment, x + y = 12 *)
let _ = x + y

(* p occurs freely here.
   This declaration cannot run unless
   p is bound in the outer environment. *)
let q (x : int) = 3 + p x

Functions are tuples

To capture this in one piece of data, OCaml represents every function value as a triple:

function = (argument, body, environment)

The argument is the parameter name, the body is the code to run, and the environment records the meaning of every identifier the body uses freely. This triple is the closure. Whenever you call the function later, the body is evaluated in the saved environment extended with the parameter binding.

In the example, let a = 2 * 7 binds a to 14. Then let p x = x + a creates a function whose body refers freely to a. The closure stored under p is the triple (x, x + a, [a := 14]). Even if a goes out of scope later, p still carries its own copy of the binding.

closure.ml 
let a = 2 * 7
let p (x : int) = x + a

(* p is bound to the closure
     ( x ,           argument
       x + a ,       body
       [a := 14] )   environment   *)

Watch a closure form

Step through this small program to see how each let grows the environment and how each function value carries its own snapshot of it.

let a = 2 * 7
let p (x : int) = x + a
let q (x : int) = x + p x
Environment
a := 14
p := (x, x + a, [a := 14])
q := (x, x + p x, [a := 14, p := (x, x + a, [a := 14])])
Step 1. Evaluate 2 * 7 in the empty environment. Bind a to 14.
Step 2. Build a closure for p. Its body uses a freely, so [a := 14] is captured.
Step 3. Build a closure for q. Its body uses both a and p freely, so both bindings are captured.
Step 4. The full environment now binds three names. Each function value carries the snapshot it needs.

Recursive functions

For a recursive function, the body refers freely to the function's own name. So the closure has to include that name too. OCaml stores recursive functions as a quadruple:

recursive function = (function identifier, argument, body, environment)

The function identifier is what makes the body able to call itself later. Without it, the recursive call would resolve as a free name with no binding.

For let rec f (i : int) : int = if i < 0 then f (i + 1) else i, the closure is (f, i, if i < 0 then f (i + 1) else i, []). The environment is empty because the body has no other free names. The only one, f, is already provided by the function-identifier slot.

Why the slot

If you tried to store a recursive function the same way as a non-recursive one, you would have to put the closure inside its own environment, which is a chicken-and-egg problem. The dedicated identifier slot avoids the cycle.

recursive.ml 
let rec f (i : int) : int =
  if i < 0 then f (i + 1) else i

(* f is bound to the closure
     ( f ,                    fn identifier
       i ,                    argument
       if i < 0 then ... ,    body
       [] )                   environment   *)
Why it matters

Closures are why returning a function from a function works in OCaml. The returned function carries the bindings it needs, even after the surrounding scope is gone. This is the mechanism behind partial application, callbacks, and most higher-order patterns.
