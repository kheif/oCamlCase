Derived Forms

Some pieces of OCaml syntax look special but are really shorthand for something simpler. The compiler rewrites them into a small core of forms before doing anything else. Knowing the rewrites makes the language feel smaller and explains a few quirks.

Boolean operators are if in disguise

The short-circuit operators && and || are not built-in functions. They are syntactic sugar for conditional expressions:

e1 && e2  →  if e1 then e2 else false
e1 || e2  →  if e1 then true else e2

This is why both operators short-circuit: the right-hand side is the body of an if branch, and an if only evaluates the branch it actually takes. There is no special evaluation rule to learn, since the rule for if already covers it.

The rewrite also explains the type. Both branches of an if must agree, so the right operand of && has to be a bool just like the literal false on the other branch. The same goes for ||.

Key point

Short-circuiting is not a separate language feature in OCaml. It is a free consequence of expressing && and || as conditionals.

short_circuit.ml 
(* Test if a predicate p returns true
   for every value in m..n *)
let rec for_all p m n =
  m > n || (p m && for_all p (m + 1) n)

(* The recursive call only fires when
   m <= n AND (p m) holds, because
   || and && short-circuit. *)

if x > 0 then (y > 0) else false
x = 3, y = 5  ⇒  if true then true else false  ⇒  true
Step 1. The source program. && looks atomic but is shorthand.
Step 2. The compiler rewrites it. The right operand becomes the then branch; false becomes the else branch.
Step 3. Substitute concrete values and the if reduces normally. No special rule for && was ever needed.

Operators as functions

Wrap any operator in parentheses and it becomes a regular two-argument function. (+) is the same as fun x y -> x + y. This works for arithmetic, comparison, and even mod.

The wrapped form is useful when an operator needs to be passed as a value: to List.fold_left, partially applied, or stored in a data structure. Without the parentheses, OCaml's parser would try to treat + as a binary operator and fail.

The unary minus has its own form, (~-), to distinguish it from binary subtraction.

Pattern

This is one of the reasons fun exists in the language at all: it is the underlying form that operators desugar to. Anywhere a function is expected, (+) works exactly like any other function value.

operators.ml 
(* Each line is a derived form:
   the right side is what it stands for *)
(-)    (* fun x y -> x - y   *)
(~-)   (* fun x   -> - x     *)
( * )  (* fun x y -> x * y   *)
(<)    (* fun x y -> x < y   *)
(=)    (* fun x y -> x = y   *)
(mod)  (* fun x y -> x mod y *)

(* Use them like any other function *)
List.fold_left (+) 0 [1; 2; 3]
(* = 6 *)
Why it matters

Once you know which forms are derived, the language has a much smaller core to remember. Conditional shorthand, operator wrapping, and even let f x = ... for let f = fun x -> ... are all rewrites on top of a tiny set of primitives. The cheat sheet shrinks; the consistency grows.
