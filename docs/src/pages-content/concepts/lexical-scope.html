Lexical Scope

A precise vocabulary for talking about names in a program: which occurrences introduce a name, which use it, which are bound to a definition, and which are free. This is the foundation for safely substituting and transforming code.

Defining vs. using occurrences

Every appearance of a name in a program is one of two kinds. A defining occurrence is the spot that introduces the name. The parameter list of a fun, the left-hand side of a let, and the variables in a pattern are all defining occurrences. A using occurrence is any other appearance, where the program reads the value that some defining occurrence stands for.

In fun x -> fun y -> z x (y x) the x after the first fun and the y after the second are defining occurrences. Every later x and y is a using occurrence that refers back to one of those parameters.

Key point

"Lexical" means the meaning of a name is fixed by where it is written, not by what happens at runtime. You can resolve every occurrence by reading the source.

occurrences.ml 
(* defining: x, y
   using:    x, y, x  *)
fun x -> fun y -> z x (y x)

(* In a let, the name on the left is defining;
   uses on the right of `in` refer back to it. *)
let n = 3 in n + n
(* defining: n
   using:    n, n  *)

Bound and free occurrences

A defining occurrence is always considered bound. A using occurrence is bound when it can be matched with a defining occurrence that is in scope, and free when no such defining occurrence exists. The act of pairing each using occurrence with the defining occurrence that supplies its value is called the lexical binding of the phrase.

In the same example, every x and y binds back to one of the fun parameters, so they are all bound. The z never gets defined inside the phrase, so it is a free occurrence. Free names depend on the surrounding context to give them meaning.

bound-vs-free.ml 
(* bound: x, y, x, y, x
   free:  z              *)
fun x -> fun y -> z x (y x)

(* This phrase has no defining occurrence
   at all, so a and b are both free.        *)
a + b

Sanitization

A phrase is called sanitized when its names are clean enough to reason about without renaming. Two conditions must hold:

  1. No identifier has more than one defining occurrence.
  2. If an identifier has a defining occurrence in the phrase, it does not also have a free occurrence in the phrase.

The first rule rules out shadowing inside the phrase, so every using occurrence has an unambiguous origin. The second rule prevents an outer free x from being silently captured by an inner fun x -> ... when code is substituted or moved around.

The first example is sanitized. The names x and y are each introduced once, and z only ever appears free, never also as a defining occurrence here.

The second example fails both conditions. x is defined by the outer let and again by the inner fun x ->. y is defined by two separate fun y -> abstractions, and it also appears free as the final argument. Before reasoning about such a phrase the inner binders should be alpha-renamed so each name is unique.

sanitization.ml 
(* Sanitized: x, y defined once; z only free. *)
fun x -> fun y -> z x (y x)

(* Not sanitized:
   - x is defined twice (outer let, inner fun)
   - y is defined twice and also appears free *)
let x = (fun y -> fun y -> (fun x -> z x y) x) y

(* Alpha-rename to clean it up: *)
let x0 = (fun y1 -> fun y2 -> (fun x1 -> z x1 y2) y1) y
Why it matters

Sanitization is the precondition for almost every program transformation: substitution, beta reduction, inlining, or refactoring a name. When a phrase is not sanitized, renaming the inner binders first prevents accidental capture and preserves the original meaning.
