List Operations

Map, filter, exists, and for_all are the workhorses of list processing. Each one is a recursive function that follows the same two-case structure as length or append, but captures a distinct idea about what to do with the elements.

Functional vs imperative

The difference between functional and imperative code is sharpest when transforming a list. Consider doubling every element of [1; 2; 3]. The functional version is a single expression. The imperative version allocates an array, loops over indices, writes to a second array, then formats the output manually.

The functional version says what to compute. The imperative version says how to compute it. The difference is not just brevity; the functional version is also easier to compose, test, and reason about because it has no side effects and produces no intermediate mutable state.

(* OCaml: one expression *)

Map

map f xs applies f to every element of xs and returns the results in a new list of the same length. The empty case returns an empty list. The cons case applies f to the head and conses the result onto the mapped tail.

The type says it all: map takes a function from 'a to 'b and a list of 'a, and returns a list of 'b. The input and output element types can differ.

Key point

map preserves structure. The output list has exactly the same length as the input; only the values change. This predictability is what makes it safe to reach for first.

map.ml 
let rec map f xs =
  match xs with
  | [] -> []
  | x :: xr -> (f x) :: map f xr

(* val map : ('a -> 'b) -> 'a list -> 'b list *)

map (fun x -> x * 2) [1; 2; 3]
(* [2; 4; 6] *)

map String.length ["hi"; "there"]
(* [2; 5] *)

Filter

filter f xs returns the list of elements from xs for which the predicate f returns true. The empty case is an empty list. The cons case includes the head if f x is true, and skips it otherwise; the tail is filtered either way.

The output list is a subsequence of the input: same order, same element type, but potentially shorter. Elements that fail the predicate simply disappear.

Key point

The type of the elements does not change through filter. It takes a 'a list and returns a 'a list. Only the count changes.

filter.ml 
let rec filter f xs =
  match xs with
  | [] -> []
  | x :: xr ->
    if f x
    then x :: filter f xr
    else filter f xr

(* val filter : ('a -> bool) -> 'a list -> 'a list *)

filter (fun x -> x >= 0) [0; -1; 2; -3; -4]
(* [0; 2] *)

Exists

exists f xs tests whether f returns true for at least one element of xs. The empty case is false: there is no element to satisfy anything. The cons case short-circuits: if f x is true, the answer is immediately true without examining the rest.

The || operator short-circuits left-to-right in OCaml, so the recursive call is only made when the head fails. On a list where a matching element appears early, almost nothing is traversed.

exists.ml 
let rec exists f xs =
  match xs with
  | [] -> false
  | x :: xr -> f x || exists f xr

(* val exists : ('a -> bool) -> 'a list -> bool *)

exists (fun x -> x < 0) [1; -2; 3]
(* true, stops at -2 *)

exists (fun x -> x < 0) [1; 2; 3]
(* false, all elements checked *)

For_all

for_all f xs tests whether f returns true for every element of xs. The empty case is true: a claim that holds for all elements of an empty list is vacuously true. The cons case short-circuits with &&: if the head fails the predicate, the answer is immediately false.

for_all and exists are exact duals. Where exists returns false for [] and uses ||, for_all returns true for [] and uses &&. The short-circuit direction also flips: for_all stops at the first element that fails.

Key point

Both functions traverse at most the whole list once. On a failing input, they stop as soon as the result is determined and never look at the remaining elements.

for_all.ml 
let rec for_all f xs =
  match xs with
  | [] -> true
  | x :: xr -> f x && for_all f xr

(* val for_all : ('a -> bool) -> 'a list -> bool *)

for_all (fun x -> x < 0) [1; -2; 3]
(* false, stops at 1 *)

for_all (fun x -> x > 0) [1; 2; 3]
(* true, all elements checked *)

Using the standard library

You now know how all four functions work internally. In practice, the List module provides them ready to use. The stdlib versions are also tail-recursive where it matters, so they handle large lists without stack overflow.

stdlib.ml 
List.map     (fun x -> x * x) [1; 2; 3; 4]
(* [1; 4; 9; 16] *)

List.filter  (fun x -> x mod 2 = 0) [1; 2; 3; 4]
(* [2; 4] *)

List.exists  (fun x -> x > 10) [1; 2; 3]
(* false *)

List.for_all (fun x -> x > 0) [1; 2; 3]
(* true *)

(* Pipe them together *)
[1; 2; 3; 4; 5]
|> List.filter  (fun x -> x mod 2 = 0)
|> List.map     (fun x -> x * x)
(* [4; 16] *)
