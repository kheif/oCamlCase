Polymorphism

When a function does not depend on the concrete type of an argument, OCaml gives it a polymorphic type. The same function then works for any type that fits the shape of its signature.

Type variables

A type written with a leading apostrophe, like 'a, is a type variable. It stands for "some type, the same one wherever this variable appears". When a function's inferred type contains type variables, the function is polymorphic.

The identity function uses its argument only by returning it. Nothing in the body pins the type, so the compiler infers 'a -> 'a: any type goes in, the same type comes out. The two occurrences of 'a are linked.

Key point

Type variables are not wildcards. 'a -> 'a means input and output have the same type. 'a -> 'b means they may differ.

poly.ml 
let identity x = x
(* identity : 'a -> 'a *)

identity 42          (* 42, used as int *)
identity "hi"        (* "hi", used as string *)
identity [1; 2; 3]   (* int list *)

(* Each use picks a concrete type
   for 'a at that call site. *)

Parametric polymorphism

OCaml's polymorphism is parametric: a polymorphic function behaves the same way for every type it could be called with. It cannot inspect the type at runtime and branch on it. This makes the type variable a real promise: the compiler enforces that the body works uniformly.

This is what allows so much of the standard library to be written once and used with any element type. List.length works on a list of integers, a list of strings, or a list of records, with no overloading and no runtime dispatch.

The signatures advertise this directly. List.length is 'a list -> int: it works on a list of any element type and returns an integer. List.rev is 'a list -> 'a list: the element type is preserved.

stdlib.ml 
List.length [1; 2; 3]
(* 3; 'a is int here *)

List.length ["a"; "b"]
(* 2; 'a is string here *)

List.rev [1; 2; 3]
(* [3; 2; 1], type int list -> int list *)

let swap (a, b) = (b, a)
(* swap : 'a * 'b -> 'b * 'a *)

Polymorphism with higher-order functions

Polymorphic types come into their own with higher-order functions. List.map has type ('a -> 'b) list... (more precisely, ('a -> 'b) -> 'a list -> 'b list). Two type variables make the relationship explicit: it takes a function from one element type to another and a list of the input type, and returns a list of the output type.

map.ml 
(* List.map : ('a -> 'b) -> 'a list -> 'b list *)

List.map String.length ["hi"; "world"]
(* [2; 5] with 'a = string, 'b = int *)

List.map (fun x -> x * x) [1; 2; 3]
(* [1; 4; 9] with 'a = int, 'b = int *)

Generalization and the value restriction

OCaml only generalizes a binding to a polymorphic type when the right-hand side is a value: a function literal, a constant, or another value-shaped expression. Bindings whose right-hand side performs computation, especially with mutable cells, get a less general type to keep the type system sound.

You usually do not have to think about this: ordinary let f x = ... definitions are always values (they are functions) and become polymorphic without effort. The restriction shows up mainly with partial applications that involve ref.

generalize.ml 
(* Function definition: fully polymorphic *)
let id x = x
(* id : 'a -> 'a *)

(* Value: polymorphic *)
let empty = []
(* empty : 'a list *)

(* Computation involving a ref:
   not generalized, type variable becomes weak *)
let store = ref []
(* store : '_weak1 list ref *)
Good practice

Let polymorphism happen on its own. Write functions in their most natural form, and the compiler will give them the most general type they support. Reach for annotations only to narrow a type, never to widen one.
