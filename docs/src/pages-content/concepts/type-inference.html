Type Inference

OCaml is statically typed but you rarely write type annotations. The compiler figures out the most general type that makes your program well-typed and uses that.

Types without annotations

When you define a binding, OCaml inspects how the value is used and assigns it a type. The algorithm that drives this is Hindley-Milner inference: every expression collects constraints from the operators and functions applied to it, and the compiler solves them.

The expression x + 1 tells the compiler that x must be an int, because + on integers has type int -> int -> int. From there the type of the whole function follows.

Key point

Inference is not guessing. The compiler computes the unique most general type that is consistent with every use of every name. If no such type exists, you get a type error instead of a silent runtime failure.

infer.ml 
let incr x = x + 1
(* incr : int -> int *)

let greet name = "hi, " ^ name
(* greet : string -> string *)

let avg a b = (a +. b) /. 2.0
(* avg : float -> float -> float *)

How constraints propagate

Each operator and function used in an expression contributes a constraint. + forces its arguments to be int. +. forces them to be float. ^ forces strings. The result type of one call becomes the input type of the next.

This is why even small changes to a function body can change its inferred type. Swapping + for +. moves the whole signature from integers to floats without you touching any annotations.

OCaml does not silently coerce between numeric types. The arithmetic operator you choose determines the type, and the type of every variable in scope follows from there.

propagate.ml 
let double_int x = x + x
(* double_int : int -> int *)

let double_float x = x +. x
(* double_float : float -> float *)

(* The body picks the type. Here, comparing
   to a string forces the input to be a string. *)
let is_yes s = s = "yes"
(* is_yes : string -> bool *)

Type variables when nothing constrains

If a function body uses an argument in a way that does not pin down its type, OCaml leaves the type open. It introduces a type variable, written 'a, 'b, and so on, to stand for "any type, chosen later".

This is the foundation of polymorphism, covered next. For now, the takeaway is that the compiler does not invent a concrete type when it does not need one. It keeps the binding as general as the program allows.

generic.ml 
let identity x = x
(* identity : 'a -> 'a *)

let first (a, _) = a
(* first : 'a * 'b -> 'a *)

let const x _ = x
(* const : 'a -> 'b -> 'a *)

Annotations as a tool, not a requirement

You can write a type annotation anywhere, but you usually do not need to. Annotations are useful when you want to document intent at a module boundary, narrow an inferred type that came out more general than you wanted, or get a clearer error message in a tricky function.

annotate.ml 
(* On the binding *)
let incr : int -> int = fun x -> x + 1

(* On a parameter *)
let incr (x : int) = x + 1

(* On the result *)
let incr x : int = x + 1

(* All three produce the same type: int -> int *)
Good practice

Let the compiler infer types in the small. Add annotations at module signatures and on tricky helpers. The combination keeps everyday code clean while preserving a clear contract at the boundaries.
